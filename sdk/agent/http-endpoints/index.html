    HTTP Endpoints

    Create custom HTTP API endpoints for your agents using the @http decorator. These endpoints are automatically registered with the FastAPI server and provide RESTful access to agent capabilities.

    Endpoints run within the same request context as chat calls and can leverage skills (auth, payments, discovery, etc.). Use scopes to protect sensitive routes and prefer simple, well-typed signatures for clarity.

    Overview

    HTTP endpoints extend your agent beyond the chat completion interface, allowing direct API access to specific functions. They work alongside chat completions, tools, and other agent capabilities.

    Key Features: - Automatic FastAPI route registration - Scope-based access control - Support for all HTTP methods - Path parameters and query strings - JSON request/response handling - Error handling and status codes - Integration with existing agent tools

    Basic Usage

    Simple HTTP Endpoint

    from robutler.agents.tools.decorators import http
from robutler.agents import BaseAgent

@http("/status")
def get_status() -> dict:
    """Simple status endpoint"""
    return {
        "status": "healthy",
        "version": "2.0.0",
        "uptime": "5h 23m"
    }

agent = BaseAgent(
    name="api-agent",
    model="openai/gpt-4o",
    http_handlers=[get_status]
)

    Available at: GET /api-agent/status

    HTTP Methods

    @http("/resource", method="get")
def get_resource() -> dict:
    return {"data": "resource"}

@http("/resource", method="post")
def create_resource(data: dict) -> dict:
    return {"created": data, "id": "123"}

@http("/resource/{id}", method="put")
def update_resource(id: str, data: dict) -> dict:
    return {"updated": data, "id": id}

@http("/resource/{id}", method="delete")
def delete_resource(id: str) -> dict:
    return {"deleted": True, "id": id}

    Async Endpoints

    @http("/async-data", method="post")
async def process_async_data(data: dict) -> dict:
    """Async endpoint for long-running operations"""
    import asyncio
    await asyncio.sleep(1)  # Simulate processing

    return {
        "processed": data,
        "processing_time": "1s",
        "status": "completed"
    }

    Scope-Based Access Control

    Control who can access your endpoints using scopes:

    @http("/public", scope="all")
def public_endpoint() -> dict:
    """Anyone can access this endpoint"""
    return {"message": "Public data"}

@http("/user-data", scope="owner")
def owner_only_endpoint() -> dict:
    """Only the agent owner can access this"""
    return {"private": "owner data"}

@http("/admin", scope="admin")
def admin_endpoint() -> dict:
    """Admin users only"""
    return {"admin": "sensitive data"}

@http("/multi-scope", scope=["owner", "admin"])
def multi_scope_endpoint() -> dict:
    """Multiple scopes - owner OR admin can access"""
    return {"data": "for owners and admins"}

    Scope Hierarchy: - "all" - Public access (default) - "owner" - Agent owner only - "admin" - Admin users only - ["owner", "admin"] - Multiple scopes (OR logic)

    Parameters and Data Handling

    Query Parameters

    @http("/search")
def search_endpoint(query: str, limit: int = 10, sort: str = "relevance") -> dict:
    """Search with query parameters"""
    return {
        "query": query,
        "limit": limit,
        "sort": sort,
        "results": [f"Result {i}" for i in range(limit)]
    }

    Usage: GET /agent/search?query=python&limit=5&sort=date

    Path Parameters

    HTTP endpoints support FastAPI-style dynamic path parameters for building RESTful APIs:

    # Simple resource access
@http("/users/{user_id}")
def get_user(user_id: str) -> dict:
    """Get user by ID"""
    return {
        "user_id": user_id,
        "name": f"User {user_id}",
        "email": f"user{user_id}@example.com"
    }

# Nested resource hierarchy
@http("/users/{user_id}/posts/{post_id}")
def get_user_post(user_id: str, post_id: str) -> dict:
    """Path parameters for resource identification"""
    return {
        "user_id": user_id,
        "post_id": post_id,
        "content": f"Post {post_id} by user {user_id}"
    }

# Deep nesting with multiple parameters
@http("/users/{user_id}/posts/{post_id}/comments/{comment_id}")
def get_comment(user_id: str, post_id: str, comment_id: str) -> dict:
    """Complex nested resource access"""
    return {
        "user_id": user_id,
        "post_id": post_id,
        "comment_id": comment_id,
        "comment": f"Comment {comment_id} on post {post_id}"
    }

# Mix path parameters with query parameters
@http("/users/{user_id}/posts")
def get_user_posts(user_id: str, limit: int = 10, published: bool = True) -> dict:
    """Combine path and query parameters"""
    return {
        "user_id": user_id,        # From path
        "limit": limit,            # From query string
        "published": published,    # From query string
        "posts": [f"Post {i}" for i in range(1, limit + 1)]
    }

    Supported URL Patterns: - GET /agent/users/123 â†’ user_id="123" - GET /agent/users/123/posts/456 â†’ user_id="123", post_id="456" - GET /agent/users/123/posts/456/comments/789 â†’ user_id="123", post_id="456", comment_id="789" - GET /agent/users/123/posts?limit=5&published=true â†’ user_id="123", limit=5, published=True

    Key Features: - âœ… Automatic parameter extraction by FastAPI - âœ… Type conversion (strings, integers, booleans) - âœ… RESTful URL patterns for resource hierarchies - âœ… Mixed parameter types (path + query + body) - âœ… Clean function signatures with type hints

    Error Handling

    HTTP endpoints automatically handle errors and return appropriate status codes:

    @http("/divide")
def divide_numbers(a: float, b: float) -> dict:
    """Division with error handling"""
    if b == 0:
        raise ValueError("Division by zero not allowed")

    return {
        "result": a / b,
        "operation": f"{a} / {b}"
    }

@http("/validate")
def validate_data(email: str) -> dict:
    """Custom validation with HTTP exceptions"""
    from fastapi import HTTPException

    if "@" not in email:
        raise HTTPException(
            status_code=400, 
            detail="Invalid email format"
        )

    return {"email": email, "valid": True}

    Error Responses: - ValueError â†’ HTTP 500 with error message - HTTPException â†’ Custom HTTP status and message - Validation errors â†’ HTTP 422 with details

    Integration with Agent Tools

    HTTP endpoints can use agent tools and integrate with the broader agent ecosystem:

    from robutler.agents.tools.decorators import tool, http

@tool(scope="owner")
def calculate_score(data: str) -> float:
    """Internal tool for score calculation"""
    # Complex calculation logic
    return 0.95

@http("/score", method="post")
def score_api(data: dict) -> dict:
    """HTTP endpoint that uses internal tool"""
    # Convert dict to string for tool
    data_str = str(data)

    # Use the internal tool
    score = calculate_score(data_str)

    return {
        "data": data,
        "score": score,
        "grade": "A" if score > 0.9 else "B"
    }

agent = BaseAgent(
    name="scoring-agent",
    tools=[calculate_score],       # Available to LLM
    http_handlers=[score_api]      # Available via HTTP
)

    Direct Registration

    Register endpoints directly on agent instances using the FastAPI-style decorator:

    agent = BaseAgent(name="my-agent", model="openai/gpt-4o")

@agent.http("/health")
def health_check() -> dict:
    """Health check endpoint"""
    return {
        "status": "healthy",
        "agent": agent.name,
        "capabilities": len(agent._registered_tools)
    }

@agent.http("/config", method="get", scope="admin")
def get_config() -> dict:
    """Configuration endpoint (admin only)"""
    return {
        "name": agent.name,
        "scopes": agent.scopes,
        "skills": list(agent.skills.keys())
    }

# Also supports tools, hooks, and handoffs
@agent.tool
def quick_calc(expression: str) -> str:
    return str(eval(expression))

    Capabilities Auto-Registration

    Use the capabilities parameter to automatically register any decorated functions:

    from robutler.agents.tools.decorators import tool, http, hook, handoff

@tool(scope="owner")
def analyze_data(data: str) -> str:
    return f"Analysis: {data}"

@http("/upload", method="post")
def upload_file(file_data: dict) -> dict:
    return {"uploaded": file_data["name"], "size": "2MB"}

@http("/download/{file_id}")
def download_file(file_id: str) -> dict:
    return {"file_id": file_id, "url": f"/files/{file_id}"}

@hook("on_request", priority=5)
def log_requests(context):
    print(f"API request: {context.request_id}")
    return context

# Auto-register all decorated functions
agent = BaseAgent(
    name="file-agent",
    model="openai/gpt-4o",
    capabilities=[analyze_data, upload_file, download_file, log_requests]
)

# Results in:
# - 1 tool (analyze_data) - available to LLM
# - 2 HTTP endpoints (upload_file, download_file) - available via HTTP API
# - 1 hook (log_requests) - executes on requests

    Server Integration

    HTTP endpoints are automatically registered with the FastAPI server when agents are added:

    from robutler.server.core.app import RobutlerServer

# Create agent with HTTP endpoints
agent = BaseAgent(
    name="api-agent",
    http_handlers=[get_status, process_data]
)

# Create server
server = RobutlerServer(agents=[agent])

# Endpoints automatically available:
# GET  /api-agent/status
# POST /api-agent/process
# GET  /api-agent (agent info)
# POST /api-agent/chat/completions (chat)

    Best Practices

    Naming Conventions

    # Good: Clear, RESTful paths
@http("/users/{id}")           # Resource-based
@http("/reports/monthly")      # Hierarchical
@http("/search")               # Action-based

# Avoid: Unclear or inconsistent paths
@http("/getUserById")          # Not RESTful
@http("/do_stuff")             # Too vague

    Response Consistency

    @http("/api/data")
def get_data() -> dict:
    """Consistent response format"""
    return {
        "success": True,
        "data": {"key": "value"},
        "timestamp": "2024-01-01T12:00:00Z",
        "meta": {
            "count": 1,
            "has_more": False
        }
    }

    Error Handling

    @http("/api/process", method="post")
def process_request(data: dict) -> dict:
    """Proper error handling"""
    try:
        # Validate input
        if not data:
            raise ValueError("Empty data not allowed")

        # Process data
        result = complex_processing(data)

        return {
            "success": True,
            "result": result
        }

    except ValueError as e:
        from fastapi import HTTPException
        raise HTTPException(status_code=400, detail=str(e))

    except Exception as e:
        from fastapi import HTTPException
        raise HTTPException(status_code=500, detail="Internal processing error")

    Security Considerations

    @http("/admin/delete", method="delete", scope="admin")
def dangerous_operation(confirm: bool = False) -> dict:
    """Secure dangerous operations"""
    if not confirm:
        raise HTTPException(
            status_code=400,
            detail="Must confirm dangerous operation"
        )

    # Additional verification could include:
    # - Rate limiting
    # - Audit logging
    # - Multi-factor authentication

    return {"deleted": True, "confirmed": confirm}

    Data Export

    @http("/export/{format}")
def export_data(format: str, limit: int = 100) -> dict:
    """Export data in different formats"""
    valid_formats = ["json", "csv", "xml"]

    if format not in valid_formats:
        raise HTTPException(
            status_code=400,
            detail=f"Invalid format. Supported: {valid_formats}"
        )

    # Generate mock data
    data = [{"id": i, "value": f"item_{i}"} for i in range(limit)]

    return {
        "format": format,
        "count": len(data),
        "data": data,
        "export_time": "2024-01-01T12:00:00Z"
    }

    Troubleshooting

    Common Issues

    1. Endpoint not found (404)
    2. Check agent is registered with server
    3. Verify subpath spelling

    4. Ensure HTTP method matches

    5. Permission denied (403)

    6. Check scope configuration
    7. Verify user has required scope

    8. Review access control logic

    9. Parameter errors (422)

    10. Check function signature
    11. Verify required parameters
    12. Review data types

    Debugging

    @http("/debug")
def debug_endpoint(request: Request) -> dict:
