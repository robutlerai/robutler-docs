    V2 Server Architecture Guide

    Beta Software Notice

    Robutler is currently in beta stage. While the core functionality is stable and actively used, APIs and features may change. We recommend testing thoroughly before deploying to critical environments.

    Overview

    Robutler V2 features a completely redesigned server architecture that provides enterprise-grade performance, scalability, and OpenAI API compatibility. This guide covers the server architecture, deployment patterns, and best practices.

    Architecture Components

    Core Server Components

    graph TB
    A[Client Request] --> B[Load Balancer]
    B --> C[API Gateway]
    C --> D[Request Router]
    D --> E[Agent Manager]
    E --> F[Skill Registry]
    E --> G[Context Manager]
    E --> H[LLM Proxy]
    H --> I[OpenAI/LiteLLM]
    F --> J[Memory Skills]
    F --> K[Platform Skills]
    F --> L[Custom Skills]

    Request Processing Flow

    1. Request Ingestion: HTTP/WebSocket requests received
    2. Authentication: API key validation and user identification
    3. Routing: Dynamic agent selection based on request path
    4. Context Creation: Request context and session management
    5. Agent Execution: Skill orchestration and LLM interaction
    6. Response Generation: Streaming or batch response delivery

    Server Configuration

    Basic Configuration

    # server.yml
server:
  host: "0.0.0.0"
  port: 8000
  workers: 4

  # OpenAI API compatibility
  openai_compatibility: true

  # Request handling
  max_request_size: "10MB"
  timeout: 300  # seconds

  # Concurrency
  max_concurrent_requests: 100
  request_queue_size: 1000

# Agent configuration
agents:
  default_agent: "robutler-assistant"
  dynamic_creation: true
  cache_duration: 3600  # seconds

# Skills configuration  
skills:
  auto_discovery: true
  skill_timeout: 60  # seconds
  max_skill_depth: 10

    Environment Variables

    # Core configuration
ROBUTLER_HOST=0.0.0.0

# LiteLLM configuration

# Database (optional)

REDIS_URL=redis://localhost:6379

# Monitoring
PROMETHEUS_ENABLED=true
HEALTH_CHECK_ENABLED=true

    Deployment Patterns

    Single Instance Deployment

    # Direct Python execution
python -m robutler.server --host 0.0.0.0 --port 8000

# With configuration file
robutler serve --config server.yml

    Docker Deployment

    FROM python:3.11-slim

WORKDIR /app

# Install dependencies
COPY requirements.txt .
RUN pip install -r requirements.txt

# Copy application
COPY . .

# Health check
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:8000/health || exit 1

# Run server
EXPOSE 8000
CMD ["python", "-m", "robutler.server", "--host", "0.0.0.0", "--port", "8000"]

    Kubernetes Deployment

    apiVersion: apps/v1
kind: Deployment
metadata:
  name: robutler-server
spec:
  replicas: 3
  selector:
    matchLabels:
      app: robutler-server
  template:
    metadata:
      labels:
        app: robutler-server
    spec:
      containers:
      - name: robutler
        image: robutler:v2-latest
        ports:
        - containerPort: 8000
        env:
        - name: ROBUTLER_HOST
          value: "0.0.0.0"
        - name: ROBUTLER_PORT
          value: "8000"
        - name: OPENAI_API_KEY
          valueFrom:
            secretKeyRef:
              name: robutler-secrets
              key: openai-api-key
        resources:
          requests:
            memory: "512Mi"
            cpu: "250m"
          limits:
            memory: "2Gi"
            cpu: "1000m"
        livenessProbe:
          httpGet:
            path: /health
            port: 8000
          initialDelaySeconds: 30
          periodSeconds: 10
        readinessProbe:
          httpGet:
            path: /health
            port: 8000
          initialDelaySeconds: 5
          periodSeconds: 5

---
apiVersion: v1
kind: Service
metadata:
  name: robutler-service
spec:
  selector:
    app: robutler-server
  ports:
  - protocol: TCP
    port: 80
    targetPort: 8000
  type: LoadBalancer

    Performance Optimization

    Connection Pooling

    # Database connection pooling
DATABASE_POOL_SIZE = 20
DATABASE_POOL_TIMEOUT = 30

# Redis connection pooling  
REDIS_POOL_SIZE = 10

    Caching Strategy

    caching:
  # Agent response caching
  agent_cache:
    enabled: true
    ttl: 300  # 5 minutes
    max_size: 1000

  # Skill result caching
  skill_cache:
    enabled: true
    ttl: 600  # 10 minutes
    max_size: 500

  # LLM response caching
  llm_cache:
    enabled: true
    ttl: 3600  # 1 hour
    max_size: 100

from robutler.server import AsyncRobutlerServer

# High-performance async server
server = AsyncRobutlerServer(
    max_workers=8,
    request_timeout=300,
    max_concurrent_requests=200

# Run with uvicorn for production
# uvicorn robutler.server:app --workers 4 --host 0.0.0.0 --port 8000

    Load Balancing

    Nginx Configuration

    upstream robutler_backend {
    least_conn;
    server robutler-1:8000 weight=1 max_fails=3 fail_timeout=30s;
    server robutler-2:8000 weight=1 max_fails=3 fail_timeout=30s;
    server robutler-3:8000 weight=1 max_fails=3 fail_timeout=30s;
}

server {
    listen 80;
    server_name api.yourcompany.com;

    location / {
        proxy_pass http://robutler_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # WebSocket support
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";

        # Timeouts
        proxy_connect_timeout 60s;
        proxy_send_timeout 60s;
        proxy_read_timeout 60s;
    }
}

    HAProxy Configuration

    global
    daemon
    maxconn 4096

defaults
    mode http
    timeout connect 5000ms
    timeout client 50000ms
    timeout server 50000ms

frontend robutler_frontend
    bind *:80
    default_backend robutler_servers

backend robutler_servers
    balance roundrobin
    option httpchk GET /health
    server robutler1 robutler-1:8000 check
    server robutler2 robutler-2:8000 check
    server robutler3 robutler-3:8000 check

    Security

    API Key Management

    # Environment-based configuration
API_KEYS = {
    "prod": os.getenv("ROBUTLER_API_KEY_PROD"),
    "staging": os.getenv("ROBUTLER_API_KEY_STAGING"),
    "dev": os.getenv("ROBUTLER_API_KEY_DEV")
}

# Rate limiting per API key
RATE_LIMITS = {
    "prod": "1000/hour",
    "staging": "500/hour", 
    "dev": "100/hour"
}

    HTTPS/TLS Configuration

    security:
  tls:
    enabled: true
    cert_file: "/path/to/cert.pem"
    key_file: "/path/to/key.pem"

  cors:
    enabled: true
    allow_origins: ["https://yourapp.com", "https://admin.yourapp.com"]
    allow_methods: ["GET", "POST", "PUT", "DELETE"]
    allow_headers: ["Authorization", "Content-Type"]

  rate_limiting:
    enabled: true
    requests_per_minute: 60
    burst_size: 10

    Troubleshooting

    Common Issues

    1. High Memory Usage
    2. Check agent caching configuration
    3. Monitor skill memory consumption

    4. Review LLM response caching

    5. Slow Response Times

    6. Check database connection pool
    7. Monitor LLM API response times

    8. Review skill execution times

    9. Connection Timeouts

    10. Adjust request timeout settings
    11. Check network connectivity
    12. Monitor connection pool health

    Debug Mode

    # Enable debug logging
ROBUTLER_LOG_LEVEL=DEBUG robutler serve

# Enable request tracing
ROBUTLER_TRACE_REQUESTS=true robutler serve

# Enable performance profiling
ROBUTLER_PROFILE=true robutler serve

    Production Checklist

    • [ ] Configure proper resource limits
    • [ ] Set up health checks and monitoring
    • [ ] Configure log rotation and retention
    • [ ] Set up backup and disaster recovery
    • [ ] Configure security settings (TLS, API keys)
    • [ ] Set up load balancing and failover
    • [ ] Configure caching for performance
    • [ ] Set up alerting and notifications
    • [ ] Test scaling and performance limits
    • [ ] Document operational procedures

    Best Practices

    1. Resource Management: Set appropriate CPU and memory limits
    2. Error Handling: Implement graceful degradation and retry logic
    3. Monitoring: Set up comprehensive metrics and alerting
    4. Security: Use proper authentication and rate limiting
    5. Performance: Optimize caching and connection pooling
    6. Scalability: Design for horizontal scaling from the start
    7. Maintenance: Plan for rolling updates and zero-downtime deployments
